# IAM

```yaml
# .aws/config
[default]
region = ap-southeast-2
output = json

# .aws/credentials
[default]
aws_access_key_id = AKIA...    ## for temporary credentials generated by STS, the prefix is ASIA
aws_secret_access_key = NrcV...
```


**IAM Role**: An entity (user, application, or service) requests temporary security credentials from STS by calling an API like `AssumeRole`.
`STS` (Security Token Service) returns a set of temporary credentials: an access key ID, a secret access key, and a session token.
The access key and secret key are used to sign the request (using AWS Signature Version 4).
The session token is added as the `X-Amz-Security-Token` header:
```yaml
Authorization: AWS4-HMAC-SHA256 Credential=<access-key-id>/<credential-scope>, SignedHeaders=<headers>,Signature=<signature>
X-Amz-Security-Token: <session-token>
```
note that even for an IAM role using temporary credentials from STS, the secret access key is never sent in the request. The temporary secret access key is used locally to sign the request, but only the access key ID, session token, and signature are sent. The temporary secret access key remains confidential and is not transmitted.
STS keeps a temporary session entry something like:
```yaml
AccessKeyId: ASIAxxxx
SecretAccessKey: (stored securely)
SessionContext:  ## <-----------these info can be represented by session-token, i.e SessionToken = Sign_AWS_PrivateKey(session_metadata)
    RoleArn: arn:aws:iam::123456789012:role/MyRole
    SessionName: MySession
    Expiration: 2026-03-13T15:00:00Z
    ...

# note that these session context is signed by AWS using internal AWS private key, not SecretAccessKey
# you migh ask why aws doesn't use SecretAccessKey to encrypt SessionContext,
# if secretAccessKey is used to sign sessionContext, then aws users can tamper the role name in SessionContext to get unauthenticated priviledge
```
note that the design idea of storing and encrypting SessionContext into session token is to prevent aws receiver service to query STS again if SessionContext is stored on STS' end and you might ask what about SecretAccessKey, isn't that STS has to store SecretAccessKey in its end? the answer is no, aws uses a share credentials (like a shared database) between each service, so receiver service doesn't need to query STS for SecretAccessKey.

temporary credentials from STS are **rotated** (such as when a new session is requested or the old one expires), a new session token is generated along with the new access key ID and secret access key. The session token is unique (because Expiration will change) to each set of temporary credentials and will change every time the credentials are rotated or renewed. Each session token is valid only for its specific session and expiration period.

When an EC2 instance starts or needs new credentials, AWS calls STS on behalf of the instance to generate fresh temporary credentials. These credentials are then stored in the instance metadata service (**IMDS** runs on `169.254.169.254`). For Lambda, these credentials are injected as environment variable 

so AWS SDK automatically:
* calls IMDS
* retrieves credentials
* refreshes them

SecretAccessKey proves that the client owns the credentials.
SessionToken proves that the session itself is valid, not expired to tampered by aws users.

An example of custom IAM Role ("Custom-EC2-Role-For-S3"):
```json
{                              // <------------this is called Trust Policy
    "Version": "2012-10-17",
    "Statement": [
        {
            "Effect": "Allow",
            "Action": [
                "sts:AssumeRole"
            ],
            "Principal": {
                "Service": [
                    "ec2.amazonaws.com"
                ]
            }
        }
    ]
}
```
and this role contain a permission of `AmazonS3FullAccess`:
```json
{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Effect": "Allow",
            "Action": [
                "s3:*",
                "s3-object-lambda:*"
            ],
            "Resource": "*"
        }
    ]
}
```
so if an IAM user want to attach the "Custom-EC2-Role-For-S3" role to an EC2 instance, the IAM user need to have permission on `iam:PassRole` first, if yes, the role is attached to the EC2 instance, now EC2 insance need `sts:AssumeRole`

Think of it like this:
`iam:PassRole` = "I'm allowed to assign this role to EC2"
`sts:AssumeRole` = "the service is allowed to become this role"

## Create A VPC with Subets

1. Create VPC `10.10.0.0/16` a default main route table below is automatically created:
```bash
# main Route Table

# Destination              Target
10.10.0.0/16               local 
```

2.  Create IGW and attach the above VPC to it

3. Create two subnets, `10.10.0.0/24` in one AZ, which will be public subnet later and `10.10.1.0/24` in another AZ, which will be private subnet
both subnets' route table is the main route table above

4. Create two route tables
```bash
# demo-public Route Table

# Destination              Target
10.10.0.0/16               local 
0.0.0.0/0                  igw
```
```bash
# demo-private Route Table

# Destination              Target
10.10.0.0/16               local 
```
and assocate them with public, private subnet accordingly

5. Create A NAT, which needs to be placed in public subnet and allocated a elastic ip, note that you don't need to specify IGW duringt NAT creation as NAT will use the route table of the subnet it is placed, in this case, it is public subnet and its route table already contains an entry `0.0.0.0/0 igw`

6. add a new entry to private subnet's route table as:
```bash
# demo-private Route Table

# Destination              Target
10.10.0.0/16               local 
0.0.0.0/0                  nat     # <----------------add this entry
```



note that **no public ip address is assigned to e.g EC2 instance**, **public IP addresses are mapped to EC2's private ip, and EC2 is attached with an Elastic Network Interface (ENI) which contaions the private ip** (even though it makes you think ip address is assigned to EC2 instance directly as the choose box says "Auto-assign public IP" in EC2 Launching). An instance can have multiple ENIs. and  `IGW` assoicates the ENI's pivate ip address with public ip via translation (this process is called **static NAT**)
so when you create an EC2 instance: AWS automatically does:

* Creates a primary ENI (called eth0)
* Attaches it to the instance
* Assigns:
  Private IP (mandatory)  // ENI always own private ip address, and that is why private ip doesn't changed like public ip after restarting
  Security group(s)    👉 now you see SG is attached to ENI, not directly attached to EC2 instance
  (Optional) Public IP mapping, performed by IGW

so when you assign a public IP (auto-assign or Elastic IP address), ENI doesn't get assigned with the public ip address  AWS records something like:
```
ENI-123 (private IP: 10.0.1.5)
        ⇅
Public IP: 54.12.34.56
```
👉 This **mapping** is:
Stored in AWS internal systems
Referenced by: Internet Gateway AWS data plane (Hyperplane)
and (ENI pviate ip, Public IP) mapping is a temporary mapping as public ip can be changed after restarting, however (ENI pviate ip, Elastic IP) is permanent mapping

also note that **NAT is actually an aws full managed EC2 instance under the hood**, so you will need to create multiple NAT for multiple AZs for resilient purpose. AWS introduces a **regional NAT**

## Gateway Endpoint Vs Inteface Endpoint

* **Gateway Endpoint**: only support S3 and DynamoDB; it adds routes to route tables
```bash
# demo-private Route Table

# Destination              Target
10.10.0.0/16               local 
pl-xxxxxxx                 vpce-xxxxxxxx  # <--------gateway endpoint id
# pl mean prefix list, and it cotnains com.amazonaws.apsoutheast-2.s3
```

* **Inteface Endpoint**: doesn't modify route table, **it create an ENI attached to the subnet**, and AWS DNS will resolve public dns name to  the ENI's privatge dns name, for example:
```bash
aws sqs send-message --queue-url "https://sqs.ap-southeast-2.amazonaws.com/070648356818/myqueue" --message-body "Hi from AWS" --endpoint-url https://vpce-0b2f4ec4bbdb4771a-js2svm3r.sqs.ap-southeast-2.vpce.amazonaws.com/

# Enable private DNS name during Endpoint Creation. no need to add --endpoint-url
aws sqs send-message --queue-url "https://sqs.ap-southeast-2.amazonaws.com/070648356818/myqueue" --message-body "Hi from AWS"
```
Regardingf Inteface Endpoint's Enable private DNS name:
* Without Private DNS enabled:
Only the VPC endpoint-specific DNS name (the long vpce-... name) resolves to the private IP of the ENI. You must explicitly use this DNS name in your applications to route traffic privately.
* With Private DNS enabled:
The standard AWS service DNS name (e.g., sqs.ap-southeast-2.amazonaws.com) is also mapped to the endpoint's private IP inside your VPC, so you can use the usual AWS service name and traffic stays private.

## ALB

**ALB Idle Timeout VS ALB Connection Pooling**:

Client-side connection pool = owned and managed by the client (ALB just keep connection open)
Connection Pool (ALB to Target Group) is managed by ALB (Target Group such as EC2 instances just keep connection open)

note that alb has two setting, "http client keep alive duration" (only control Client ↔ ALB ) and "Connection idle timeout" (this is the **ALB idle timeout**, control both Client ↔ ALB AND ALB ↔ Target), "http client keep alive duration" is like the maxinum time a tcp connection can be used/reused, if Connection idle timeout is 5s and http client keep alive duration is 10s, then the TCP connection can only be reused twice. You might ask why there is no maxinum timeout setting (similar to "http client keep alive duration") for ALB ↔ Target, it is because **Backend connections are fully under your control (the server), not ALB’s responsibility**


```
👉 Once ALB sends FIN:

The connection enters half-closed state
But for HTTP usage, it is effectively unusable

because from client's perspective:

It receives FIN → knows server won’t send more data
Any new HTTP request:
❌ cannot be safely sent on this connection
So client will:
stop using it
close its side (send FIN)

👉 Result: full closure happens quickly
```


1. ALB Idle Timeout (primary)

Default:

60 seconds

👉 If no traffic flows:

Idle → timeout reached → ALB sends FIN → connection removed from pool ❌

2. Target server keep-alive timeout

Example (Node.js): `server.keepAliveTimeout = 65000`

👉 If server closes first:

Server sends FIN → connection gone ❌

so both ALB Idle Timeout and server.keepAliveTimeout control how long can an TCP connection stay in the pool/to be resued

Case: server timeout is shorter
`server.keepAliveTimeout` = 5000, 5s < ALB idle timeout 60s
t=0   request done
t=5   server sends FIN ❌
t=30  ALB tries reuse → connection already dead → ❌ error (502)
that's why we need to set  `server.keepAliveTimeout` > ALB idle timeout, i.e
```js
server.keepAliveTimeout = 65000; // ALB = 60s
```

## NLB

* unlike ALB whose note get dynamic ip, NLB get static ip (elastic ip) for each node, so clients's company can whitelist these NLB's ip addresses.
* NLB **rewrites** packets received from client by chaning dest to be target's ip and source to be itself, then this packet will be route through VPC to the target, and when response is sent back to NLB, NLB will do NAT to rewrite dest to be client's IP
* Despite NLB is a pass-through load balancer, so the target sees the client's original IP directly, but there is one exception that NLB can terminate SSL request and work like ALB, so NLB will create a new TCP connection (plain text) between NLB <--> Target, refer to the diagram to see more details, the idea is to let NLB to decrpt rather than let targets to decrypt (decrypt is CPU expensive work) and let targets to focus on app logic works


## ALB VS NLB

**ALB**: two tcp connections involved, client <--> ALB as TCP connection1, ALB <--> Target as TCP connection2, these two connections are in parallel, and ALB can buffer response from TCPConn2 and "forward" repsonse to former in TCPConn1. ALB creates an internal record to associate relavent response and client. ALB will send `X-Forwarded-For` that includes the original client's ip to target

**NLB**: only one TCP connection involved, NLB rewrites packet's dest, so (Client IP, Client Port, NLB IP, NLB Listener Port, Protocol) is mapped to a particular target, event same client and port send another request, still same target receives it,  it is like itcky Sessions always enable. Target see client IP without the need of X-Forwarded-For header

note that NLB cannot do round-robin like ALB, because NLB doesn't terminate client <--> ALB connection, ALB can do round-robin because a new TCP connection in created for ALB <--> Target, while NLB use a singe TCP connection, NLB only rewrites packet, so a unique tuple (Client IP, Client Port, NLB IP, NLB Listener Port, Protocol) can only send multiple requests to a single target e.g targetA (let's say NLB choose targetA). Contrast to ALB, unique tuple (Client IP, Client Port, ALB IP, ALB Listener Port, Protocol) can send multiple requests to multiple targets, becuase ALB create multiple TCP connections to different targets (so SYN can be send to different targets), while NLB doesn't terminate client to ALB connections and only rewrite packet, so no new TCP connections created to different targets, only a singe TCP connection is for TargetA for example, and if client sends another request (reuse client to NLB TCP connection), even though it is targetB chosen for the request, but targetB cannot receive SYN since client OS doesn't send SYN

you might ask NLB's hash(5-tuple) → target flow hash algorithm can make some clients (different client ip address) to be mapped/tied to a particular target, so there will be no load balancing, it is true and false, the statement is true but NLB's load balancing is probabilistic, with few clients, there is no load balancing, with thousands/millions client, the load balance can be achieved, like Law of large numbers in Statistics

In term of "Load Balancing", ALB seems to be active, because it can use round-robin to almost guarantee load being distributed evenly;  while NLB is passive, since it is up to the unique tuple, only the number of different client reachs to a higher number, then load is distributed *average* evenly .

**Cross-Zone Load Balancing**
says there are tGA with 2 instances in AZ-a, tGB with 4 instance in AZ-a, with CZLB off, each AZ get 50% (because client uses round-robin to query each elb node evenly, 50% 50% part is client-driven), each instance in AZ-a gets 25%, each instance in AZ-b gets 12.5%, not really balanced. With CZLB on, each AZ still get 50% (because of client-driven), but nodeA in AZ-a can send choose target in AZ-b, all each node of 6 nodes get 100%/6 = 16.67%, perfect evenly load

- ALB: CZLB is on by default, and users are supposed to pay inter-AZ data transfer fee (image you send 1GB data from AZ-a to target in AZ-b), but AWS waive it, so users don't need to pay
- NLB: CZLB is off by default, and users need to pay inter-AZ data transfer fee when they turn it on

## S3 

S3 Storage Classes:

R3AZs : replicated to 3 AZs
DRF   : data retrievals fees per GB
DRT   : data retrievals time
DT    : data transfer fees, Data Transfer In is free, Data Transfer OUT incurs fees, apply on all classes, so it won't be listed in the different classes below
NP    : no public access like S3 static website
storage fees and DT fees applys on all classes

* `S3 Standard`:                     R3AZs, immediate access, no DRF
* `S3 Standard - Infrequent Access`: R3AZs, immediate access, DRF, storage cost is half price of `S3 Standard`, but need to pay DRF
* `S3 One Zone - Infrequent Access`: 1 AZ,  immediate access, DRF, user cannot select which AZ to save object while S3 Express One Zone allows uses to choose AZ
* `S3 Glacier - Instant`:            R3AZs, immediate access, higher DRF, it save more if you access data less frequent than S3 Standard - IA
* `S3 Glacier - Flexible`:           R3AZs, DRT: users can choose `Expedited` 1 to 5 min, `Standard` 3 to 5 hours (that's why it is called Flexible), cheaper DRF, less storage fees, NP,
* `S3 Glacier - Deep Archive`:       R3AZs, DRT from 12 to 48 hours, cheapest DRF, cheapest storage fees, NP

* `Intelligent Tiering`: move objects automatically between `Frequent Access Tier`,  `Infrequent Access Tier` and `Archive Instant Access Tier`, note that e.g. Frequent Access Tier is almost the same as S3 standard, but since you avoid burden by letting aws to switch tier for you, you have to pay a little bit more for above 2 tiers. Based on usage, default tier is Frequent Access tier, then objects not accessed for 30 days are moved to Infrequent Access tier.

* `S3 Express One Zone`: highest performance in all S3 classes; 10x faster and 80% lower cost than S3 Standard, designed specifically to work with S3 Directory Buckets (a new bucket type deployed into a single Availability Zone, optimized for low-latency, high-request-rate operations)

note that **the storage class is assigned per object, not per bucket**. AWS calculates S3 storage costs on a daily basis.
